# Documentação Técnica das Funcionalidades (Frontend ↔ Supabase)

Este documento detalha funcionalidades específicas do app e como elas se conectam ao Supabase. **Todas já estão implementadas** (a tabela `debts`, o cálculo "On Fire", o deck de participantes e os refinamentos do ranking). Para o setup geral e a arquitetura, veja o [README.md](README.md).

## 1. Sistema de "Pendurar Aposta" (Fiados)
Permite que participantes pendurem a aposta diária de **R$ 2,50** na caderneta de fiados caso não queiram/possam realizar o Pix imediatamente.

### Integração no Banco de Dados (Supabase)
O frontend consome a tabela `debts` no Supabase nas funções `handleRegisterDebt`, `handleRemoveDebt` e `handleRemoveAllDebts` dentro de [App.tsx](src/App.tsx). A tabela e suas políticas de segurança (RLS: cada um só insere/remove o próprio fiado) ficam em [supabase/update-005-fiados.sql](supabase/update-005-fiados.sql), e o Realtime em [supabase/update-006-debts-realtime.sql](supabase/update-006-debts-realtime.sql).

#### Schema Recomendado para PostgreSQL (Supabase):
```sql
CREATE TABLE public.debts (
    id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id text NOT NULL,                    -- ID do participante (referencia o UID do participante/auth)
    amount numeric(10, 2) NOT NULL DEFAULT 2.50, -- Valor fixo da aposta diária (R$ 2,50)
    debt_date date NOT NULL,                  -- Data da rodada / aposta pendurada (Formato: YYYY-MM-DD)
    created_at timestamp with time zone DEFAULT timezone('utc'::text, now()) NOT NULL
);

-- Índices sugeridos para performance:
CREATE INDEX idx_debts_user_id ON public.debts(user_id);
CREATE INDEX idx_debts_date ON public.debts(debt_date);
```

#### Comportamento no Frontend:
1. **Gravação:** Quando o usuário clica em "Pendurar" (na aba Financeira/Pix), o frontend faz um `INSERT` na tabela `debts` passando o `user_id` do participante e a data de hoje formatada no fuso de Brasília (`debt_date`).
2. **Exclusão (Dar baixa):** Ao clicar no ícone de "remover/lixo" ao lado do fiado pendurado, é executado um `DELETE` filtrando por `id` da dívida.
3. **Fallback Local:** Caso o Supabase falhe ou a tabela `debts` ainda não exista, o frontend salva os dados automaticamente no `localStorage` sob a chave `bolao_local_debts` e exibe mensagens de aviso no console.

## 2. Sistema "On Fire" (Streak de Pontuação)
Identifica e exibe de forma destacada (entre o ranking geral e a lista de participantes) se algum participante está em uma sequência "On Fire" de pontuação consecutiva.

### Regra de Negócio e Cálculo:
O status é calculado dinamicamente no frontend na função `getFireCounts` em [StandingsTable.tsx](src/components/StandingsTable.tsx) com as seguintes condições:
1. Busca todos os jogos finalizados (`status === 'finished'` com placares preenchidos).
2. Ordena os jogos cronologicamente pelo horário de kickoff.
3. Filtra apenas os jogos finalizados **em que o participante de fato realizou um palpite** (evitando que jogos que o usuário não apostou quebrem sua sequência).
4. Verifica as últimas **5 apostas válidas** do participante.
5. Se o participante pontuou (`points > 0` na função `analyzeBet`) em **todas** essas últimas 5 apostas, ele entra em modo **"ON FIRE"**.

### Integração com o Backend:
* Atualmente, o cálculo é feito em tempo de execução no cliente a partir da lista de palpites (`bets`) e partidas (`matches`).
* **Se desejado no futuro:** O backend pode expor um campo booleano `is_on_fire` no perfil do usuário, calculado periodicamente via trigger no banco de dados ou funções serverless sempre que uma partida for dada como concluída.

## 3. Carrossel de Participantes Empilhado (Layout Deck 3D)
O slideshow rotativo e a visualização em grade colapsável foram substituídos por um **deck interativo de fotos empilhadas 3D** em [StandingsTable.tsx](src/components/StandingsTable.tsx).

* **Visual:** As fotos estão empilhadas no centro com deslocamento e rotação gradual, simulando um monte físico de cartas. O cabeçalho é estático (não colapsável) e o visual é extremamente polido.
* **Interação:** Suporta gestos de arrasto (drag/swipe) com o mouse ou toque.
  * Deslizar para a esquerda ou clicar avança para o próximo participante.
  * Deslizar para a direita retrocede.
  * Controles de dots de paginação na parte inferior atualizam o indicador ativo.
* **Ajustes:** O badge textual `"PARTICIPANTE"` foi removido, focando apenas no nome do jogador. O tamanho dos cartões foi aumentado para `240px` de largura por `315px` de altura para dar visibilidade total.

## 4. Refinamentos Visuais no Ranking
Realizamos ajustes finos nos componentes e estilos do ranking para valorizar todas as posições:

* **Medalhas Flutuantes (1º e 2º colocado):** As medalhas de Ouro e Prata agora transbordam a borda superior direita dos cards (`top: -12px; right: -12px`). Elas possuem uma máscara de transparência no CSS baseada nas próprias imagens WebP (`-webkit-mask-image`) e recebem uma animação de sweep brilhante metálico síncrona com os números de pontuação.
* **Melhorias no 3º e 4º Lugar:**
  * **3º Lugar:** Recebeu fundo gradiente em bronze, bordas na cor bronze (`rgba(205, 127, 50, 0.28)`) e badge de colocação circular bronze personalizado.
  * **4º Lugar:** Recebeu fundo em gradiente cinza escuro, bordas suaves de contorno e badge circular clássico de contorno destacado.
